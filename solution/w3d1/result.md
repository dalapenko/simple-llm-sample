# Тематика
Демонстрация трёхслойной модели памяти (Layered Memory): изоляция слоёв Short-Term, Work Memory и Long-Term, удержание фактов при переполнении скользящего окна.

**Запись сессии**: [`layered-memory.cast`](layered-memory.cast) (asciinema v3, 120×35)

---

## Сценарий тестирования

**Тема**: Проверка удержания фактов при полном вытеснении стартовых сообщений из окна короткой памяти.
**Модель**: GPT-4o Mini
**Стратегия**: Layered Memory (`--strategy layered --context-window 4`)

Ключевой вопрос: смогут ли факты, сохранённые в **Work** и **Long-Term** слоях, пережить полное переполнение скользящего окна диалога (где они уже давно выпали бы в `Sliding Window`)?

### Ход тестирования:

```
1. /memory add long-term   Меня зовут Дмитрий, я пишу на Kotlin
2. /memory add work        Текущая задача: реализация Layered Memory
3. Привет! Помоги разобраться с DI в Kotlin
4. А какой фреймворк лучше выбрать, Koin или Dagger?
5. Расскажи про плюсы Koin в 3 пунктах
6. Напомни, как меня зовут, и над какой задачей я сейчас работаю?   ← проверка памяти
7. /memory list                                                       ← инспекция слоёв
8. /exit
```

При `--context-window 4`, к моменту проверочного вопроса (шаг 6) три предшествующих обмена (#3–#5) полностью заняли окно краткосрочного диалога. Факты из шагов #1–#2 давно выпали бы из него при использовании `Sliding Window`.

## Распределение данных по слоям

### Short-Term (краткосрочные заметки)
Слой для ручных заметок с ограниченным скользящим окном. В данном тесте заполнялся через `/memory add short-term` (не использовался) — остался пустым.
Данные в этом слое: **ручные быстрые заметки** — вещи, о которых нужно помнить прямо сейчас, но не в долгосрочной перспективе. Пример: «remember: use snake_case in this file».

### Work Memory (рабочая память)
Слой для данных **текущей задачи**. Живёт в рамках одного запуска приложения.
В тесте сохранено: `Текущая задача: реализация Layered Memory` — сигнал для LLM о том, чем занят пользователь.

### Long-Term (долгосрочная память)
Слой для **профильных фактов** (имя, стек, правила проекта). Персистируется в `~/.llmchat/long-term-memory.json` — сохраняется между перезапусками.
В тесте сохранено: `Меня зовут Дмитрий, я пишу на Kotlin`.

## Output (результаты взаимодействия)

### Проверка памяти (шаг 6 — после переполнения окна)

**Input:**
```
Напомни, как меня зовут, и над какой задачей я сейчас работаю?
```

**Output:**
```

```

**Итог**: Модель **успешно вспомнила оба факта**, несмотря на то что окно диалога (ctx) было полностью занято тремя предыдущими обменами про DI/Koin. Факты сохранились в Work и Long-Term слоях и инжектируются в системный промпт независимо от скользящего окна.

## Статистика токенов

Команды `/memory add` не инициируют LLM-вызовы, поэтому статистика начинается с первого текстового сообщения.

| Ход | Событие / Сообщение | input | ctx (окно) | work | long | output | total |
|:---:|---------------------|:-----:|:----------:|:----:|:----:|:------:|:-----:|
| 1 | `/memory add long-term ...` | — | — | — | — | — | — |
| 2 | `/memory add work ...` | — | — | — | — | — | — |
| 3 | «Привет! Помоги разобраться с DI в Kotlin» | ~10 | ~0 | ~10 | ~9 | ~342 | ~371 |
| 4 | «А какой фреймворк лучше выбрать, Koin или Dagger?» | ~12 | ~352 | ~10 | ~9 | ~170 | ~553 |
| 5 | «Расскажи про плюсы Koin в 3 пунктах» | ~8 | ~534 | ~10 | ~9 | ~143 | ~704 |
| **6** | **«Напомни, как меня зовут...» (проверка памяти)** | ~15 | **~685** | ~10 | ~9 | **~19** | ~738 |

**Наблюдения по токенам:**

- `work` (~10) и `long` (~9) — константа на **каждом** шаге. Пользователь сам контролирует их размер через `/memory add` / `/memory delete`.
- `ctx` растёт по мере наполнения окна (шаги 3 → 4 → 5 → 6), затем стабилизируется у предела `context-window = 4`.
- На проверочном вопросе (шаг 6) ctx был большим (~685 токенов под 4 обмена), но этот контекст — только обсуждение DI/Koin. Факты об имени и задаче пришли из `work` и `long` — поэтому ответ точный.

## Влияние слоёв на ответы ассистента

### Механизм инжекции в системный промпт

`LayeredMemoryStrategy.buildContextBlock()` формирует секции в системном промпте в следующем порядке:

```
[LONG-TERM KNOWLEDGE]
• [c71e8e4d] Меня зовут Дмитрий, я пишу на Kotlin

[CURRENT TASK]
• [10a43f91] Текущая задача: реализация Layered Memory

[RECENT CONVERSATION] (last 4 of max 4 exchanges)
User: Привет! Помоги разобраться с DI в Kotlin
Assistant: ...
...
```

Порядок имеет значение: **Long-Term → Work → Short-Term Notes → Conversation**. LLM читает промпт последовательно, и контент, расположенный ближе к концу системного промпта (т.е. вся история диалога), получает больший «вес» для следования стилю. Но факты, явно размеченные заголовками `[LONG-TERM KNOWLEDGE]` и `[CURRENT TASK]`, остаются устойчивыми маркерами независимо от позиции.

### Почему Sliding Window потерял бы факты

В `Sliding Window` с `--context-window 4` к шагу 6 в окне остались бы только обмены #3–#5 (три вопроса про DI). Обмены #1–#2 из окна исчезли бы. Но в Layered Memory шаги #1–#2 были командами `/memory add`, а не обменами чата, — факты записались в отдельные персистентные слои.

## Вывод

### Качество и стабильность
Стратегия **Layered Memory** демонстрирует безупречную стабильность удержания фактов. В отличие от `Sticky Facts`, где LLM-экстрактор может случайно перезаписать или утерять факт при очередном ходе, здесь факт гарантированно присутствует в промпте до тех пор, пока пользователь явно не удалит его (`/memory delete`) или не очистит слой (`/memory clear`).

### Эффективность (токены)
Стратегия максимально прозрачна и предсказуема:
- Токены расходуются **только на то, что пользователь сам решил сохранить**.
- Стоимость дополнительного контекста линейна и полностью управляема.
- Отсутствуют скрытые LLM-вызовы (как в `Sticky Facts`, где на каждый ход требовался дополнительный вызов экстрактора).

### Управляемость
- Пользователь имеет полный (`/memory list`) и точечный (`/memory delete`, `/memory clear`) контроль.
- Структурированные заголовки (`[LONG-TERM KNOWLEDGE]`, `[CURRENT TASK]`) помогают самой LLM понимать вес и природу каждого факта лучше, чем безликий поток сообщений в истории.
- Long-Term память персистируется на диске → факты выживают между перезапусками.

### Сравнение со стратегиями из W2D5

| Критерий | Sliding Window | Sticky Facts | Branching | **Layered Memory** |
|---|:---:|:---:|:---:|:---:|
| Удержание фактов вне окна | ✗ | ⚠ нестабильно | ✓ (в пределах окна) | ✓ гарантировано |
| Предсказуемость расхода токенов | ✓ | ⚠ (скрытые вызовы) | ✗ растёт | ✓ |
| Персистентность | ✗ | ✗ | ✗ | ✓ Long-Term |
| Контроль пользователя | ✗ | частичный | команды веток | ✓ явный по слоям |
| Дополнительные LLM-вызовы | нет | +1/ход | нет | нет |

**Итог**: Layered Memory — идеальный баланс между явным контролем, предсказуемыми расходами и надёжным удержанием фактов. Требует ручных действий (`/memory add`), но взамен обеспечивает 100% гарантию сохранности архитектурных решений, текущих задач и профильных настроек пользователя.
