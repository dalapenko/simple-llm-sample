# MCP (Model Context Protocol) — внешние инструменты для LLM-агента

## Что такое MCP и зачем это нужно

Обычный LLM умеет только генерировать текст. Он не может прочитать файл, сделать запрос к API или выполнить поиск — без явного подключения инструментов. Раньше каждый такой инструмент нужно было описывать вручную в промпте и реализовывать самому.

**MCP (Model Context Protocol)** — открытый стандарт, который решает эту проблему. Любая внешняя система (файловая система, база данных, браузер, GitHub) может «упаковать» свои возможности в MCP-сервер. Агент подключается к нему один раз и сразу получает готовый список инструментов с полными описаниями — что умеет, какие параметры принимает, что возвращает.

---

## Как это работает

```
CLI → /mcp connect npx ... → запускает MCP-сервер как процесс
                            ↓
              сервер объявляет свои инструменты
                            ↓
              LLM видит инструменты и может их вызывать
                            ↓
              сервер выполняет запрос и возвращает результат
                            ↓
              LLM формирует ответ пользователю
```

Пользователь спрашивает «покажи файлы в папке» — агент сам решает вызвать инструмент `list_directory`, получает реальный список и отвечает на основе него, а не придумывает.

---

## Руководство по использованию

### 1. Подключение к серверу

```
/mcp connect <команда> [аргументы...]
```

После подключения CLI запустит MCP-сервер, получит список его инструментов и передаст их агенту. С этого момента LLM может их использовать в разговоре.

**Примеры готовых серверов:**

```
# Файловая система — читать, писать, искать файлы
/mcp connect npx -y @modelcontextprotocol/server-filesystem .

# Веб — загружать и читать страницы по URL
/mcp connect npx -y @modelcontextprotocol/server-fetch

# Git — просматривать историю, коммиты, диффы
/mcp connect npx -y @modelcontextprotocol/server-git .

# GitHub — работать с репозиториями, issues, PR
/mcp connect npx -y @modelcontextprotocol/server-github
```

> Для работы с `npx`-серверами нужен Node.js. При первом запуске сервер скачивается автоматически.

---

### 2. Посмотреть доступные инструменты

```
/mcp tools
```

Покажет список всего, что умеет подключённый сервер: названия инструментов, описания и параметры. Так можно понять, о чём вообще можно спрашивать агента.

---

### 3. Проверить статус подключения

```
/mcp status
```

---

### 4. Отключиться

```
/mcp disconnect
```

После отключения LLM снова работает только с текстом — без инструментов.

---

### 5. Что спрашивать после подключения

После того как сервер подключён, просто общайтесь с агентом естественным языком — он сам решит, когда вызывать инструменты:

```
# С filesystem-сервером:
Покажи все файлы в текущей директории
Прочитай файл README.md
Найди все .kt файлы в папке src

# С fetch-сервером:
Загрузи страницу https://example.com и скажи, о чём она
```

## Плюсы и минусы MCP

### Плюсы

**Готовая экосистема.** Существуют десятки серверов: filesystem, fetch, git, postgres, sqlite, puppeteer, github, notion, slack и другие. Подключение занимает одну команду — писать код не нужно.

**LLM сам решает, когда звонить.** Не нужно составлять сложные промпты вида «используй инструмент X для задачи Y». Агент видит описания инструментов и сам понимает, когда и что вызвать.

**Изоляция.** Каждый сервер — отдельный процесс. Файловый сервер видит только ту папку, которую ему передали. Это ограничивает область воздействия при ошибке.

**Единый протокол.** Один и тот же CLI подключается к файловой системе, базе данных или внешнему API — без изменений в коде.

### Минусы

**Требует Node.js** для большинства готовых серверов (они написаны на TypeScript). При первом запуске `npx -y` скачивает сервер — нужен интернет и несколько секунд ожидания.

**Агент может ошибаться с выбором инструмента.** LLM не всегда правильно интерпретирует описание инструмента или передаёт корректные аргументы. Это особенность LLM-инструментов в целом, не специфика MCP.

**Нет персистентности между сессиями.** При каждом запуске CLI нужно заново подключаться к серверу командой `/mcp connect`.

**Только stdio-транспорт.** Текущая реализация поддерживает только локальные серверы, запущенные как дочерний процесс. Удалённые HTTP/SSE серверы не поддерживаются.
