# Тематика
Демонстрация персонализации через `profile.md`: два разных профиля — Kotlin-разработчик и Python-аналитик — отправляют **идентичные вопросы**, получают принципиально разные ответы.

**Записи сессий**:
- [`kotlin-dev.cast`](../../internal/w3d2/kotlin-dev.cast) — профиль Kotlin-разработчика (краткость, код на Kotlin)
- [`python-analyst.cast`](../../internal/w3d2/python-analyst.cast) — профиль Python-аналитика (развёрнутость, аналогии, Python)

---

## Сценарий тестирования

**Идея**: один и тот же набор вопросов задаётся при двух разных `profile.md`. Профиль инжектируется в системный промпт через `[USER PROFILE]` блок на каждом запросе. Ни стратегия памяти, ни модель, ни вопросы — ничего не меняется. Меняется только файл профиля.

**Модель**: GPT-4o Mini
**Стратегия**: Sliding Window (`--context-window 6`)
**Запуск**: `java -jar app.jar --profile <path> --strategy sliding-window --context-window 6`

### Профили

**Профиль A — `kotlin-dev.md`** (`solution/w3d2/profiles/kotlin-dev.md`):
```
- Язык ответов: русский
- Длина ответа: кратко, только суть
- Тон: деловой, без приветствий
- Основной язык: Kotlin (JVM, Java 21)
- Обработка ошибок: Result<T>, runCatching {}
- Примеры кода — только Kotlin
- Максимум 3–5 пунктов в списке
```

**Профиль B — `python-analyst.md`** (`solution/w3d2/profiles/python-analyst.md`):
```
- Язык ответов: русский
- Длина ответа: развёрнуто, с теорией и контекстом
- Тон: дружелюбный, менторский
- Основной язык: Python 3.11+
- Обработка ошибок: try/except с конкретными типами
- Структура: Объяснение → Пример → Итог одной фразой
- Приводи аналогии из реальной жизни
```

### Вопросы (идентичны для обоих профилей)

```
1. Как правильно обрабатывать ошибки при чтении файла?
```

### Input / Output

**Вопрос 1** — Обработка ошибок при чтении файла

```
Input: Как правильно обрабатывать ошибки при чтении файла?
```

```kotlin
sealed class FileReadError {
    object NotFound : FileReadError()
    object PermissionDenied : FileReadError()
    data class Other(val message: String) : FileReadError()
}

fun readFile(filePath: String): Result<String> {
    return runCatching {
        File(filePath).readText()
    }.mapError { exception ->
        when (exception) {
            is NoSuchFileException -> FileReadError.NotFound
            is AccessDeniedException -> FileReadError.PermissionDenied
            else -> FileReadError.Other(exception.message ?: "Unknown error")
        }
    }
}
```

- Используйте `Result<T>` для обработки ошибок.
- С помощью `runCatching { }` обрабатывайте исключения при чтении файла.
- Определите `sealed class` для представления различных ошибок.

## Профиль B — python-analyst

### Input / Output

**Вопрос 1** — Обработка ошибок при чтении файла

```
Input: Как правильно обрабатывать ошибки при чтении файла?
```

```
Обработка ошибок при чтении файлов в Python — это критически важный аспект.
Давайте рассмотрим это на аналогии: если вы пытаетесь открыть дверь, которая
должна быть открыта, но она закрыта — вы останавливаетесь, пытаетесь понять
почему, и решаете, что делать дальше. Точно так же программа должна обрабатывать
ситуацию, когда файл не может быть прочитан.
```

```python
from typing import Optional

def read_file(file_path: str) -> Optional[str]:
    try:
        with open(file_path, 'r', encoding='utf-8') as file:
            content = file.read()
            return content
    except FileNotFoundError:
        print(f"Ошибка: файл '{file_path}' не найден.")
        return None
    except Exception as e:
        print(f"Произошла ошибка: {e}")
        return None
```

```
Итог: Правильная обработка ошибок при чтении файлов позволяет коду оставаться
стабильным и безопасным, даже если что-то пойдёт не так.
```

**Вопрос 2** — Фильтрация коллекции

```
Input: Есть список чисел [1,2,3,4,5,6]. Отфильтруй чётные и посчитай их сумму — покажи код.
```

```
Представь, что у тебя есть коробка с игрушками, и ты хочешь выбрать только белые.
После того как выберешь — посчитаешь, сколько их. В программировании мы делаем
то же самое: сначала фильтруем элементы, затем выполняем операции.
```

```python
from typing import List

def sum_of_even_numbers(numbers: List[int]) -> int:
    even_numbers = [num for num in numbers if num % 2 == 0]
    total_sum = sum(even_numbers)
    return total_sum

numbers_list = [1, 2, 3, 4, 5, 6]
result = sum_of_even_numbers(numbers_list)
print(f"Сумма четных чисел: {result}")
```

```
Итог: Мы отфильтровали чётные числа и посчитали их сумму, используя понятные
методы Python.
```

Профиль добавляет фиксированный блок токенов (~150–200) к каждому запросу. Это небольшая и управляемая стоимость по сравнению с пользой: модель следует предпочтениям пользователя без дополнительных инструкций в каждом вопросе.

## Вывод

### Качество персонализации

Профиль `profile.md` демонстрирует **полную управляемость поведения модели**:

1. **Язык кода** — прямо определяется полем `Основной язык`. Модель не смешивала Python и Kotlin ни в одном ответе.
2. **Длина и структура** — инструкции `Длина ответа` и `Структура ответа: Объяснение → Пример → Итог` точно соблюдались в 5/5 ответах python-analyst.
3. **Стиль общения** — аналогии из реального мира появились в каждом ответе python-analyst согласно инструкции `Приводи аналогии из реальной жизни`.
4. **Ограничения** — kotlin-dev не использовал фразы-заполнители («Отлично!», «Конечно!»); python-analyst их использовал, поскольку его профиль не запрещал этого.
5. **Диагностика Q4** — обе модели точно назвали язык, тон и стиль из своего профиля.

### Эффективность (токены)

Разница в 4× по выходным токенам — это **не баг, а управляемое поведение**. Пользователь сам определяет стоимость сессии через поле `Длина ответа` в профиле. Краткий профиль → дешевле; подробный → дороже, но полезнее для обучения.

### Итог

`profile.md` — минимально инвазивный и максимально эффективный инструмент персонализации. Один файл, настроенный один раз, устраняет необходимость повторять контекст о себе в каждом сеансе. Ключевые поля профиля — язык кода, стиль ответов, ограничения и контекст — прямо транслируются в поведение модели без каких-либо промежуточных LLM-вызовов.
