# Тематика
Демонстрация **Task State Machine FSM**: сериализуемый конечный автомат для отслеживания длительных задач. CLI сохраняет состояние задачи при паузе и восстанавливает его при следующем запуске — LLM продолжает работу без повторного объяснения контекста.

**Запись сессии**:
- [`demo.gif`](./demo.gif) — две последовательные сессии: старт задачи → пауза → перезапуск → продолжение без переинтро

---

## Сценарий тестирования

**Идея**: в рамках одной записи показаны два независимых запуска CLI. Первый сеанс создаёт задачу, проводит её через состояния и сохраняет при выходе. Второй сеанс восстанавливает задачу и продолжает разговор с LLM без единого слова о том, что именно делается.

**Модель**: GPT-4o Mini
**Стратегия**: Sliding Window (`--context-window 6`)
**Запуск**: `java -jar app.jar --strategy sliding-window --context-window 6`

### Задача демо

```
Реализовать функцию поиска пользователей в базе данных
```

### Шаги сессии 1 — старт и пауза

```
1. /task start Реализовать функцию поиска пользователей в базе данных
2. /task status                          ← проверка начального состояния (PLANNING)
3. [LLM] Какой подход лучше использовать для поиска по имени и email: LIKE или полнотекстовый поиск?
4. /task advance execution               ← переход PLANNING → EXECUTION
5. /task step Пишем SQL-запрос с индексом
6. /task pause                           ← сохранение в ~/.llmchat/task-state.json
   /exit
```

### Шаги сессии 2 — продолжение без переинтро

```
7. (CLI запускается) → баннер "Resuming Task" → ответ "y"
8. [LLM] Покажи пример SQL-запроса для поиска пользователя по частичному совпадению имени
         ← вопрос без контекста "у меня есть задача...", LLM отвечает в тему
9. /task status                          ← подтверждение что состояние живо
10. /task done                           ← попытка завершить → ошибка FSM (EXECUTION → DONE запрещён)
    /exit
```

Шаг 10 намеренно демонстрирует защиту FSM от пропуска этапов.

## Сессия 1 — начало задачи

### Статус после `/task start`

```
 Task Status
──────────────────────────────────────────────────
  ID:          ca0a3367…
  Task:        Реализовать функцию поиска пользователей в базе данных
  Stage:       Planning
  Step:        Initial planning
  Next action: LLM Generation
──────────────────────────────────────────────────
```

### Вопрос к LLM (шаг планирования)

```
Input: Какой подход лучше использовать для поиска по имени и email:
       LIKE или полнотекстовый поиск?
```

```
Для поиска по имени и email лучше использовать полнотекстовый поиск. Он:

• Эффективнее для обработки больших объёмов текстовой информации.
• Позволяет производить более сложные запросы (поиск по нескольким словам,
  с учётом их значимости).
• Гарантирует лучшую производительность и точность при больших наборах данных.

LIKE может быть использован в простых случаях, но имеет ограниченную
производительность и неэффективен при больших объёмах.
```

### Переход и пауза

```
/task advance execution
  → Task: Planning → Execution

/task step Пишем SQL-запрос с индексом

/task pause
  → Task paused and saved. It will be offered for resume on next startup.
```

Состояние записывается в `~/.llmchat/task-state.json`:

```json
{
  "id": "ca0a3367-...",
  "taskDescription": "Реализовать функцию поиска пользователей в базе данных",
  "stage": "EXECUTION",
  "currentStep": "Пишем SQL-запрос с индексом",
  "expectedAction": "LLM_GENERATION",
  "createdAt": 1749123456789,
  "updatedAt": 1749123489012,
  "history": [
    { "fromStage": "PLANNING", "toStage": "EXECUTION", "step": "", "timestamp": 1749123480000 }
  ]
}
```

## Сессия 2 — продолжение

### Баннер восстановления

При следующем запуске CLI видит файл `task-state.json` и показывает баннер **до** основного приветствия:

```
 ⟳ Resuming Task
──────────────────────────────────────────────────
  Task:  Реализовать функцию поиска пользователей в базе данных
  Stage: Execution
  Step:  Пишем SQL-запрос с индексом
──────────────────────────────────────────────────

 Resume task? [y/N]: y
```

После ответа `y` объект `TaskFSM` создаётся из сохранённого состояния и передаётся в `ConversationManager.setTaskFsm()`. С этого момента каждый запрос к LLM содержит в системном промпте блок `[ACTIVE TASK]`:

```
[ACTIVE TASK]
Task: Реализовать функцию поиска пользователей в базе данных
Stage: EXECUTION (Execution)
Current Step: Пишем SQL-запрос с индексом
Expected Next Action: LLM_GENERATION

You are resuming this task exactly where it left off. Do not ask the user to recap.
Continue from the current step. The conversation history above provides context.
[END ACTIVE TASK]
```

### Вопрос к LLM без переинтро ✅

```
Input: Покажи пример SQL-запроса для поиска пользователя
       по частичному совпадению имени
```

```
Вот пример SQL-запроса для поиска пользователя по частичному совпадению имени:

  SELECT *
  FROM users
  WHERE name LIKE '%часть_имени%';

Здесь '%часть_имени%' замените на ту часть имени, которую вы ищете.
Символы % обозначают, что перед и после искомой части могут быть любые символы.
```

LLM **не спросил** «что за задача?» и **не попросил** повторить контекст. Ответ содержит таблицу `users` и поле `name` — точно те сущности, которые обсуждались в предыдущей сессии через блок задачи.

### Защита FSM от неверного перехода ✅

```
/task done
 ERROR Invalid transition: EXECUTION → DONE.
       Allowed from EXECUTION: VALIDATION, PLANNING, ERROR
```

FSM отверг попытку перейти сразу в `DONE` минуя этап `VALIDATION`. Это ожидаемое поведение — машина состояний защищает от пропуска обязательных шагов рабочего процесса.

| Сессия | Вопрос                         | input | ctx | output | total |
|:------:|--------------------------------|:-----:|:---:|:------:|:-----:|
|   1    | Подход к поиску (LIKE vs FTS)  |  ~22  | ~0  |  ~110  | ~132  |

## Логика работы конечного автомата

### Граф переходов

```
PLANNING ──→ EXECUTION ──→ VALIDATION ──→ DONE
   ↑             │               │
   │             └──→ PLANNING ←─┘
   │             └──→ ERROR  ←──┘
   └──────────────────────────────
ERROR ──→ PLANNING
```

### Допустимые переходы

|     Из \ В     | PLANNING | EXECUTION | VALIDATION | DONE | ERROR |
|:--------------:|:--------:|:---------:|:----------:|:----:|:-----:|
|  **PLANNING**  |    —     |     ✅     |     —      |  —   |   ✅   |
| **EXECUTION**  |    ✅     |     —     |     ✅      |  —   |   ✅   |
| **VALIDATION** |    ✅     |     ✅     |     —      |  ✅   |   ✅   |
|    **DONE**    |    —     |     —     |     —      |  —   |   —   |
|   **ERROR**    |    ✅     |     —     |     —      |  —   |   —   |

`DONE` — терминальное состояние, переходов нет.

### Ключевые принципы

**1. Сериализация через `@Serializable`**

Весь граф состояния (`TaskState`) — `@Serializable` data class. Сохранение и восстановление: один вызов `Json.encodeToString` / `decodeFromString`, никакой ручной маппинг. Файл: `~/.llmchat/task-state.json`.

**2. `Result<TaskState>` из `transition()`**

Переход возвращает `Result<T>`, не бросает исключение. Обработчик в `Main.kt` использует `fold {}` — успех отображает переход, ошибка показывает сообщение без падения REPL. Демо шага 10 (`EXECUTION → DONE`) показывает это в действии.

**3. История переходов**

`TaskState.history: List<TaskTransitionRecord>` накапливает каждый переход с меткой времени. Это позволяет видеть, по какому пути задача прошла, даже после нескольких сессий.

**4. Блок `[ACTIVE TASK]` в системном промпте**

`TaskFSM.buildResumptionContext()` генерирует текстовый блок, который `ConversationManager.buildSystemPrompt()` вставляет после профиля и перед историей. LLM читает его на каждом запросе в сессии. Стоимость: фиксированные ~60–80 токенов на запрос, независимо от длины истории.

**5. Автосохранение через shutdown hook**

```kotlin
Runtime.getRuntime().addShutdownHook(Thread {
    taskFsm?.let { fsm ->
        if (fsm.getState().stage != TaskStage.DONE) {
            TaskStateStorage.save(fsm.getState())
        }
    }
})
```

Ctrl+C, `kill`, завершение процесса — состояние всегда сохраняется, если задача не `DONE`. Явный `/task pause` перед выходом делает то же самое явно.

## Зачем это нужно

### Проблема без FSM

Без механизма сохранения состояния при каждом перезапуске CLI пользователь вынужден:
1. Вспоминать, на каком этапе остановился
2. Пересказывать LLM контекст: «я делаю X, сейчас на шаге Y, до этого мы обсудили Z»
3. Тратить несколько ходов на «входной блок», прежде чем получить полезный ответ

Это критично для длительных задач (разработка фичи, дебаг, рефакторинг), которые занимают несколько рабочих сессий.

### Что даёт FSM

| Без FSM                      | С FSM                                              |
|------------------------------|----------------------------------------------------|
| Контекст теряется при выходе | Состояние сохраняется в JSON                       |
| Нужно пересказывать задачу   | LLM получает `[ACTIVE TASK]` автоматически         |
| LLM может уйти не туда       | Переходы валидируются; `EXECUTION → DONE` запрещён |
| Нет истории шагов            | `history[]` фиксирует весь путь с метками времени  |
| Ctrl+C = потеря прогресса    | Shutdown hook сохраняет при любом завершении       |

### Демо как доказательство

В записи сессии 2 пользователь печатает **«Покажи пример SQL-запроса»** — ни слова о задаче, таблице, контексте. LLM отвечает с `users`, `name`, `LIKE` — точно в теме. Единственный источник этого знания — блок `[ACTIVE TASK]` в системном промпте. Без FSM этот ответ был бы невозможен.

## Вывод

Task FSM превращает одноразовые чат-сессии в непрерывный рабочий процесс. Конечный автомат:

1. **Фиксирует**, где находится задача (`stage`, `currentStep`)
2. **Защищает** от неверных переходов (нельзя перейти в `DONE` без `VALIDATION`)
3. **Восстанавливает** контекст для LLM без участия пользователя
4. **Переживает** перезапуски, Ctrl+C и любые завершения процесса

Ключевой инсайт: блок `[ACTIVE TASK]` инжектируется в системный промпт, не в историю — он не уходит из контекстного окна, не суммируется и не вытесняется. Задача всегда «видна» LLM на протяжении всей сессии.
