# MCP-пайплайн — три инструмента в одном сервере

## Что сделано

Мы реализовали один MCP-сервер (`mcp-server-pipeline`), который предоставляет три инструмента сразу: `search`, `summarize` и `save_to_file`. Один `/mcp connect` достаточно, чтобы агент получил полный инструментарий для сценария «найти → обработать → сохранить». Порядок вызовов определяет LLM самостоятельно, опираясь на описания инструментов.

---

## Архитектура

```
 Пользователь                CLI (cli-app)
      │                            │
      │  /mcp connect pipeline     │    ┌──────────────────────────────────────┐
      │──────────────────────────▶ │───▶│  mcp-server-pipeline                 │
      │                            │    │                                      │
      │  [LLM] найди ... → сохрани │    │  ┌──────────┐  ┌───────────┐        │
      │──────────────────────────▶ │    │  │  search  │  │ summarize │        │
      │                            │    │  └──────────┘  └───────────┘        │
      │     ╔═══════════════╗      │    │        ┌──────────────┐             │
      │     ║ LLM (агент)   ║      │    │        │ save_to_file │             │
      │     ║               ║      │    │        └──────────────┘             │
      │     ║ 1. search()   ║      │    └──────────────────────────────────────┘
      │     ║ 2. summarize()║      │
      │     ║ 3. save_to_file()║   │  ToolRegistry = search + summarize + save_to_file
      │     ╚═══════════════╝      │  (все три из одного подключения)
```

**Сервер** — самостоятельный JAR-файл, запускается как подпроцесс и общается с CLI по протоколу JSON-RPC 2.0 через stdin/stdout. При инициализации возвращает список из трёх инструментов в одном ответе на `tools/list`.

**CLI** получает все три инструмента при первом же подключении и передаёт их агенту как единый реестр. Агент видит три инструмента и сам решает порядок их вызова.

---

## Инструменты пайплайна

| Инструмент     | Что делает                                                                       |
|----------------|----------------------------------------------------------------------------------|
| `search`       | Поиск по DuckDuckGo Instant Answers — возвращает структурированный JSON          |
| `summarize`    | Краткий пересказ любого текста или JSON через LLM (OpenRouter)                   |
| `save_to_file` | Сохраняет контент в файл в `~/.llmchat/pipeline-results/`                        |

Описания инструментов содержат явные подсказки для LLM о порядке вызовов:
- `search` → «*После вызова передай результат в `summarize`*»
- `summarize` → «*После вызова передай результат в `save_to_file`*»
- `save_to_file` → «*Финальный шаг пайплайна*»

---

## Почему это может быть полезно

**Простота развёртывания.** Один JAR — одна зависимость. Для запуска пайплайна достаточно одной команды `/mcp connect` вместо трёх. Меньше соединений, меньше подпроцессов, проще конфигурация.

**Логически связанные инструменты вместе.** Если инструменты всегда используются как группа и не имеют смысла по отдельности — их естественно держать в одном месте. Сервер оформляет эту связность явно: одна кодовая база, одна точка обслуживания.

**Агент как оркестратор.** Логика вызова инструментов не зашита в код — её выстраивает LLM на основе описаний схем. Добавить новый шаг пайплайна означает добавить описание инструмента, а не менять оркестратор.

**Стандартный протокол.** MCP — открытый стандарт. Сервер можно подключить к любому совместимому клиенту: Claude Desktop, Cursor, другому CLI-инструменту.

## Как это работает внутри

### Регистрация инструментов

При `/mcp connect` CLI запускает JAR как подпроцесс и обменивается инициализационными сообщениями по JSON-RPC. В ответ на `tools/list` сервер возвращает массив из трёх схем:

```
tools/list response:
  [
    { "name": "search",       "inputSchema": { ... } },
    { "name": "summarize",    "inputSchema": { ... } },
    { "name": "save_to_file", "inputSchema": { ... } }
  ]
```

Все три инструмента попадают в `ToolRegistry` агента за одно подключение.

### Маршрутизация вызовов

Когда LLM решает вызвать инструмент, CLI отправляет `tools/call` с именем:

```
{"method": "tools/call", "params": {"name": "search", "arguments": {...}}}
```

Сервер маршрутизирует по полю `name` к нужному обработчику — `executeSearch`, `executeSummarize` или `executeSaveToFile`. Все три живут в одном процессе и используют общие клиенты.

### Видимость вызовов в терминале

CLI интегрирован с Koog Tracing — каждый вызов инструмента отображается в терминале до и после выполнения:

```
  → [MCP] search(query="Kotlin programming language")
  ✓ [MCP] search → ...
  → [MCP] summarize(content="...", maxLength="300")
  ✓ [MCP] summarize → ...
  → [MCP] save_to_file(filename="kotlin-pipeline.json", ...)
  ✓ [MCP] save_to_file → ...
```

### Безопасность `save_to_file`

Сервер принимает только имена файлов без разделителей пути. Все файлы сохраняются строго в `~/.llmchat/pipeline-results/`. Перед записью выполняется канонизация пути и проверка, что результирующий путь остаётся внутри разрешённой директории.

## Руководство по использованию

### 1. Сборка сервера и CLI

```bash
./gradlew :cli-app:shadowJar :mcp-server-pipeline:shadowJar
```

### 2. Запуск CLI

```bash
./start.sh
# или
./gradlew :cli-app:run --console=plain
```

### 3. Подключение сервера

```
/mcp connect java -jar mcp-server-pipeline/build/libs/mcp-server-pipeline-1.0-SNAPSHOT-all.jar
```

CLI отобразит панель подключения с тремя доступными инструментами.

### 4. Просмотр инструментов

```
/mcp tools
```

Отображает все три инструмента из подключённого сервера.

### 5. Запуск пайплайна одним запросом

```
Найди информацию о языке программирования Kotlin, сделай краткое резюме и сохрани результат в файл kotlin-pipeline.json
```

→ LLM последовательно вызывает все три инструмента через один сервер:
- `search("Kotlin programming language")` — получает данные из DuckDuckGo
- `summarize(searchResults, maxLength="300")` — создаёт краткое резюме
- `save_to_file("kotlin-pipeline.json", summary)` — сохраняет результат

Файл появляется в `~/.llmchat/pipeline-results/kotlin-pipeline.json`.

### 6. Отключение

```
/mcp disconnect
```

## Плюсы и минусы

### Плюсы

**Один JAR — полный пайплайн.** Единственная команда `/mcp connect` даёт доступ ко всем инструментам. Меньше шагов при настройке.

**Общий контекст выполнения.** Все инструменты работают в одном JVM-процессе: можно делить кэш, конфигурацию, клиентские соединения — повторная инициализация не нужна.

**Единая точка обслуживания.** Один модуль, одна зависимость, одно место для логов и диагностики.

**LLM сам выстраивает цепочку.** Оркестрация задаётся описаниями инструментов, а не жёстким кодом. Агент самостоятельно определяет порядок и передаёт данные между шагами.

### Минусы

**DuckDuckGo Instant Answers ограничен.** API хорошо работает для известных сущностей (языки, технологии, люди), но не для произвольных запросов.

**Данные между шагами передаёт LLM.** Агент копирует вывод одного инструмента во вход следующего. При большом объёме данных это расходует контекстное окно.

**Нет встроенного retry.** Если шаг `summarize` завершился ошибкой, `save_to_file` не вызовется. Восстановление остаётся на усмотрение LLM.

## Структура модуля

```
mcp-server-pipeline/
├── build.gradle.kts
└── src/main/kotlin/mcppipeline/
    ├── Main.kt              # точка входа
    ├── McpServer.kt         # JSON-RPC цикл + маршрутизация трёх инструментов
    ├── DuckDuckGoClient.kt  # HTTP-клиент DuckDuckGo (инструмент search)
    ├── OpenRouterClient.kt  # HTTP-клиент OpenRouter (инструмент summarize)
    └── FileManager.kt       # безопасная запись файлов (инструмент save_to_file)
```

### Переменные окружения

| Переменная           | Обязательная       | Описание                                                              |
|----------------------|--------------------|-----------------------------------------------------------------------|
| `OPENROUTER_API_KEY` | Да (для summarize) | API-ключ для вызовов LLM через OpenRouter                             |
| `PIPELINE_MODEL`     | Нет                | Модель для суммаризации (по умолчанию: `google/gemini-2.0-flash-001`) |
