# MCP-пайплайн — три независимых сервера как единая цепочка инструментов

## Что сделано

Мы разбили задачу «найти → обработать → сохранить» на три отдельных MCP-сервера, каждый из которых отвечает ровно за одно действие. Когда все три подключены к CLI одновременно, агент видит три инструмента и сам выстраивает цепочку их вызовов — никакого специального кода для оркестрации не нужно.

Ключевое изменение в CLI: теперь `/mcp connect` **добавляет** сервер к пулу активных соединений вместо того, чтобы заменять предыдущее. Все реестры инструментов объединяются через `ToolRegistry.plus()` — агент получает плоский список инструментов из всех подключённых серверов.

---

## Архитектура

```
 Пользователь                CLI (cli-app)
      │                            │
      │  /mcp connect search       │    ┌─────────────────────────────┐
      │──────────────────────────▶ │───▶│  mcp-server-search          │
      │  /mcp connect summarize    │    │  инструмент: search          │
      │──────────────────────────▶ │    └─────────────────────────────┘
      │  /mcp connect save-file    │    ┌─────────────────────────────┐
      │──────────────────────────▶ │───▶│  mcp-server-summarize       │
      │                            │    │  инструмент: summarize       │
      │  [LLM] найди ... → сохрани │    └─────────────────────────────┘
      │──────────────────────────▶ │    ┌─────────────────────────────┐
      │                            │───▶│  mcp-server-save-file        │
      │                            │    │  инструмент: save_to_file    │
      │     ╔═══════════════╗      │    └─────────────────────────────┘
      │     ║ LLM (агент)   ║      │
      │     ║               ║      │  ToolRegistry = search ⊕ summarize ⊕ save_to_file
      │     ║ 1. search()   ║      │
      │     ║ 2. summarize()║      │
      │     ║ 3. save_to_file()║   │
      │     ╚═══════════════╝      │
```

**Каждый сервер** — это самостоятельный JAR-файл, который запускается как подпроцесс и общается с CLI по протоколу JSON-RPC 2.0 через stdin/stdout. Серверы ничего не знают друг о друге.

**CLI** объединяет реестры инструментов всех подключённых серверов в один список и передаёт его агенту. Агент видит три инструмента и сам решает, в каком порядке их вызывать, опираясь на описания в схемах.

---

## Серверы пайплайна

| Сервер                 | Инструмент     | Что делает                                                                             |
|------------------------|----------------|----------------------------------------------------------------------------------------|
| `mcp-server-search`    | `search`       | Поиск по DuckDuckGo Instant Answers — возвращает структурированный JSON с результатами |
| `mcp-server-summarize` | `summarize`    | Краткий пересказ любого текста или JSON через LLM (OpenRouter)                         |
| `mcp-server-save-file` | `save_to_file` | Сохраняет контент в файл в `~/.llmchat/pipeline-results/`                              |

Описания инструментов содержат явные подсказки для LLM о порядке вызовов:
- `search` → «*После вызова передай результат в `summarize`*»
- `summarize` → «*После вызова передай результат в `save_to_file`*»
- `save_to_file` → «*Финальный шаг пайплайна*»

---

## Почему это может быть полезно

**Разделение ответственности на уровне процессов.** Каждый сервер делает одно и умеет это хорошо. Если нужно заменить способ поиска — меняется только `mcp-server-search`, остальные не трогаются.

**Гибкость подключения.** Три сервера — это не жёсткая связка. Можно подключить только `search` и `save_to_file`, минуя суммаризацию. Или подключить `summarize` к данным из другого источника. Пайплайн складывается из подключённых серверов, а не задаётся жёстко в коде.

**Агент как оркестратор.** Не нужно писать логику «вызови A, передай результат в B, затем в C». Эту работу делает LLM — именно для этого он обучен выстраивать последовательность инструментов по их описаниям.

**Стандартный протокол.** MCP — открытый стандарт. `mcp-server-search` можно подключить к любому другому MCP-клиенту: Claude Desktop, Cursor, другому CLI-инструменту.

## Как это работает внутри

### Множественные подключения в CLI

До этой задачи CLI поддерживал только одно MCP-соединение: при `/mcp connect` предыдущий сервер завершался. Теперь `McpConnectionManager` хранит список активных соединений:

```
connections: List<ActiveConnection>
    ├─ mcp-server-search   (process + registry + stderr reader)
    ├─ mcp-server-summarize (process + registry + stderr reader)
    └─ mcp-server-save-file (process + registry + stderr reader)
```

При каждом `/mcp connect` новый сервер **добавляется** к пулу. Объединённый реестр:

```kotlin
val merged = connections.fold(ToolRegistry.EMPTY) { acc, conn -> acc + conn.registry }
```

`ToolRegistry.plus()` из фреймворка Koog объединяет инструменты по именам (дубликаты исключаются), возвращая единый плоский список.

### Безопасность `save_to_file`

Сервер принимает только имена файлов без разделителей пути — никаких `../` или абсолютных путей. Все файлы сохраняются строго в `~/.llmchat/pipeline-results/`. Перед записью выполняется канонизация пути и проверка, что результирующий путь остаётся внутри разрешённой директории.

## Руководство по использованию

### 1. Сборка всех серверов и CLI

```bash
./gradlew :cli-app:shadowJar \
          :mcp-server-search:shadowJar \
          :mcp-server-summarize:shadowJar \
          :mcp-server-save-file:shadowJar
```

### 2. Запуск CLI

```bash
./start.sh
# или
./gradlew :cli-app:run --console=plain
```

### 3. Подключение трёх серверов

```
/mcp connect java -jar mcp-server-search/build/libs/mcp-server-search-1.0-SNAPSHOT-all.jar
/mcp connect java -jar mcp-server-summarize/build/libs/mcp-server-summarize-1.0-SNAPSHOT-all.jar
/mcp connect java -jar mcp-server-save-file/build/libs/mcp-server-save-file-1.0-SNAPSHOT-all.jar
```

После каждого подключения CLI показывает количество инструментов от нового сервера и суммарное число инструментов из всех серверов.

### 4. Просмотр инструментов

```
/mcp tools
```

Отображает все три инструмента из всех подключённых серверов в едином списке.

### 5. Запуск пайплайна одним запросом

```
Найди информацию о языке программирования Kotlin, сделай краткое резюме и сохрани результат в файл kotlin-summary.json
```
→ LLM последовательно вызывает все три инструмента:
- `search("Kotlin programming language")` — получает данные из DuckDuckGo
- `summarize(searchResults, maxLength="500")` — создаёт краткое резюме
- `save_to_file("kotlin-summary.json", summary)` — сохраняет результат

Каждый вызов отображается в терминале благодаря интеграции с Koog Tracing:
```
  → [MCP] search(query="Kotlin programming language")
  ✓ [MCP] search → ...
  → [MCP] summarize(content="...", maxLength="500")
  ✓ [MCP] summarize → ...
  → [MCP] save_to_file(filename="kotlin-summary.json", ...)
  ✓ [MCP] save_to_file → ...
```

Файл появляется в `~/.llmchat/pipeline-results/kotlin-summary.json`.

### 6. Отключение

```
/mcp disconnect
```
Останавливает все подключённые серверы одновременно.

## Плюсы и минусы

### Плюсы

**Каждый сервер — одна функция.** Код поиска не знает о коде суммаризации. Легко тестировать, заменять, расширять по отдельности.

**Пайплайн конфигурируется подключениями.** Не нужно переписывать код, чтобы изменить цепочку — достаточно подключить другой набор серверов.

**LLM сам выстраивает последовательность.** Описания инструментов направляют агента в нужном порядке. Оркестрация вынесена в естественный язык, а не в императивный код.

**Стандартный протокол.** Любой MCP-клиент может подключиться к этим серверам без изменений.

### Минусы

**DuckDuckGo Instant Answers ограничен.** API хорошо работает для известных сущностей (языки, технологии, люди), но не для произвольных запросов. Составные фразы вроде «Kotlin coroutines» могут не дать результатов.

**Данные между шагами передаёт LLM.** Агент сам копирует вывод одного инструмента во вход следующего. При большом объёме данных это расходует контекстное окно.

**Нет состояния между серверами.** Если шаг `summarize` завершился с ошибкой, `save_to_file` не вызовется — но никакого встроенного механизма retry или rollback нет. Восстановление остаётся на усмотрение LLM.

## Структура модулей

```
mcp-server-search/
├── build.gradle.kts
└── src/main/kotlin/mcpsearch/
    ├── Main.kt              # точка входа
    ├── McpServer.kt         # JSON-RPC цикл + инструмент search
    └── DuckDuckGoClient.kt  # HTTP-клиент DuckDuckGo

mcp-server-summarize/
├── build.gradle.kts
└── src/main/kotlin/mcpsummarize/
    ├── Main.kt
    ├── McpServer.kt         # JSON-RPC цикл + инструмент summarize
    └── OpenRouterClient.kt  # HTTP-клиент OpenRouter

mcp-server-save-file/
├── build.gradle.kts
└── src/main/kotlin/mcpsavefile/
    ├── Main.kt
    ├── McpServer.kt         # JSON-RPC цикл + инструмент save_to_file
    └── FileManager.kt       # безопасная запись файлов
```

### Переменные окружения

| Переменная           | Обязательная       | Описание                                                              |
|----------------------|--------------------|-----------------------------------------------------------------------|
| `OPENROUTER_API_KEY` | Да (для summarize) | API-ключ для вызовов LLM через OpenRouter                             |
| `PIPELINE_MODEL`     | Нет                | Модель для суммаризации (по умолчанию: `google/gemini-2.0-flash-001`) |
