# Тематика

Демонстрация **Invariant Management System**: механизм управления жёсткими, ненарушаемыми ограничениями проекта. CLI хранит инварианты отдельно от истории диалога, инжектирует их в системный промпт перед каждым запросом, принуждая LLM проверять каждый ответ на соответствие правилам и отказывать с явной ссылкой на конкретный инвариант при нарушении.

**Запись сессии**:
- [`demo.gif`](./demo.gif) — одна непрерывная сессия: добавление ограничений → нормальный ответ → нарушение с отказом → redo с исправлением ID → удаление → свободный ответ

---

## Сценарий тестирования

**Идея**: в рамках одной записи демонстрируется полный жизненный цикл инварианта: создание, применение (блокировка запрещённого ответа), ошибочная попытка удаления, redo с корректным ID, удаление ограничения и подтверждение снятия блокировки.

**Модель**: GPT-4o Mini
**Стратегия**: Sliding Window (`--context-window 8`)
**Запуск**: `java -jar app.jar --strategy sliding-window --context-window 8`

### Инварианты демо

```
[INV-740434] [Technology Stack]   Использовать только Kotlin для backend, Java и Python запрещены
[INV-47BF90] [Architecture]       Запрещено использовать глобальные изменяемые переменные
```

### Шаги демо

```
 1. /invariant add --category stack Использовать только Kotlin для backend, Java и Python запрещены
 2. /invariant add --category architecture Запрещено использовать глобальные изменяемые переменные
 3. /invariant list                        ← отображение активных ограничений
 4. [LLM] Напиши функцию на Kotlin для вычисления числа Фибоначчи
          ← нормальный ответ, нарушений нет
 5. [LLM] Перепиши ту же функцию на Java  ← нарушение INV-740434
          → ⚠ INVARIANT VIOLATION: [INV-740434] (Technology Stack)
 6. /invariant remove INV-000000           ← намеренно неверный ID → ошибка
    ↺ REDO: "Неверный ID — смотрим список и исправляем"
 7. /invariant list                        ← видим реальные ID, скрипт извлекает INV-740434
 8. /invariant remove INV-740434           ← удаляем stack-ограничение по правильному ID
 9. /invariant list                        ← осталась только architecture
10. /invariant clear                       ← удаляем все оставшиеся
11. /invariant list                        ← список пуст
12. /clear                                 ← очищаем историю (свежий контекст)
13. [LLM] Напиши функцию на Java для вычисления факториала числа
          ← свободный ответ, ограничений нет
```

## Нормальный ответ — шаг 4 (Kotlin, без нарушений)

### Input:

```
Напиши функцию на Kotlin для вычисления числа Фибоначчи
```

### Output:

```kotlin
fun fibonacci(n: Int): Long {
    require(n >= 0) { "n must be non-negative." }
    return when (n) {
        0 -> 0
        1 -> 1
        else -> {
            var a = 0L
            var b = 1L
            repeat(n - 1) {
                val tmp = a + b
                a = b
                b = tmp
            }
            b
        }
    }
}
```

LLM свободно ответил кодом на Kotlin. Инварианты присутствуют в системном промпте — LLM проверил их и не нашёл нарушений: технология соответствует, глобальных переменных нет.

### Statistics

```
[in: ~13 | ctx: ~0 | out: ~90 | total: ~103]
```

`ctx: ~0` — первый запрос в сессии, история пуста.

### Input:

```

```

### Output:


LLM **отказался генерировать код** и процитировал конкретный инвариант по ID. Никакого Java-кода в ответе нет. Отказ структурирован: содержит ID правила, категорию и текст ограничения.

### Statistics

```
[in: ~7 | ctx: ~103 | out: ~54 | total: ~164]
```

`ctx: ~103` — в контексте находится предыдущий ответ на Kotlin-вопрос. `out: ~54` — отказ лаконичен: LLM не тратит токены на объяснения, а только цитирует правило.

## Redo — шаги 6–11 (исправление неверного ID)

### Попытка удаления с неверным ID:

```

```

```
  [INV-740434] Использовать только Kotlin для backend, Java и Python запрещены

  Architecture:
  [INV-47BF90] Запрещено использовать глобальные изменяемые переменные
```

Скрипт автоматически извлёк `INV-740434` из вывода и подставил правильный ID в следующую команду.

```

```

### Список после удаления stack:

```

```

```java
public class FactorialCalculator {
    public static long factorial(int n) {
        if (n < 0) {
            throw new IllegalArgumentException("Число должно быть неотрицательным");
        }
        long result = 1;
        for (int i = 2; i <= n; i++) {
```

## Статистика токенов

| Запрос                            | Тип          | input | ctx  | output | total |
|:----------------------------------|:-------------|:-----:|:----:|:------:|:-----:|
| Kotlin Фибоначчи (норма)          | норма        |  ~13  |  ~0  |  ~90   |  ~103 |
| Java Фибоначчи (нарушение)        | нарушение    |  ~7   | ~103 |  ~54   |  ~164 |
| Java факториал (после снятия)     | норма        |  ~13  |  ~0  | ~160   |  ~173 |

**Ключевые наблюдения:**
- Отказ (`out: ~54`) в 3× короче свободного ответа (`out: ~160`) — LLM не тратит токены на объяснения при нарушении
- `ctx: ~0` у запроса после `/clear` подтверждает: инварианты живут в **системном промпте**, не в истории — даже чистая история не возвращает инварианты в контекст
- `ctx: ~103` у нарушения — это токены предыдущего Kotlin-ответа в окне контекста

## Технический механизм

### Структура хранения инварианта

```kotlin
@Serializable
data class Invariant(
    val id: String,              // "INV-740434" (INV- + 6 случайных hex-символов)
    val description: String,     // текст ограничения
    val category: InvariantCategory,  // STACK | ARCHITECTURE | BUSINESS_RULE | GENERAL
    val createdAt: Long
)
```

Файл хранения: `~/.llmchat/invariants.json`. Формат идентичен `long-term-memory.json` и `task-state.json` — последовательная эволюция одного паттерна персистентности в проекте.

### Порядок блоков в системном промпте

```
[базовый системный промпт]

[PROJECT INVARIANTS — MANDATORY CONSTRAINTS]   ← ПЕРВЫЙ блок
  These rules are absolute and non-negotiable...
  INVARIANTS:
    Technology Stack:
      • [INV-740434] Использовать только Kotlin...

[USER PROFILE]                                 ← второй блок
[RECENT CONVERSATION]                          ← третий блок
[ACTIVE TASK]                                  ← четвёртый блок (если есть)
```

Инварианты инжектируются **первыми** после базового промпта — трансформеры обрабатывают системный промпт последовательно, и ограничения в начале получают максимальный «вес внимания» при генерации.

### Шаблон принуждения

Блок инвариантов содержит явную инструкцию:

```
Before proposing any technical solution, code, or architecture, you MUST
explicitly verify your response against every invariant listed below.

If your proposed solution conflicts with any invariant, you MUST refuse
and respond with:
  ⚠ INVARIANT VIOLATION: Cannot proceed — this conflicts with [<id>]
  (<category>): <description>.
  Please reconsider your approach or explicitly update the invariant.
```

Это **Chain-of-Thought принуждение** без явной CoT-разметки: инструкция заставляет LLM неявно выполнить проверку перед ответом, а формат отказа стандартизирован.

### Изоляция от истории

`InvariantStorage` — отдельная подсистема. Инварианты **никогда** не попадают в:
- `strategy.addMessage()` — историю диалога
- `ConversationStorage` — сохранение сессии
- контекстное окно (`ctx`)

Они всегда в `systemPrompt` и не вытесняются скользящим окном.

## CLI-команды

| Команда                                          | Что делает                                    |
|:-------------------------------------------------|:----------------------------------------------|
| `/invariant add <desc>`                          | Добавить ограничение в категорию GENERAL      |
| `/invariant add --category stack <desc>`         | Добавить ограничение в категорию STACK        |
| `/invariant add --category architecture <desc>`  | Добавить ограничение в категорию ARCHITECTURE |
| `/invariant add --category business-rule <desc>` | Добавить бизнес-правило                       |
| `/invariant list`                                | Отобразить все ограничения (по категориям)    |
| `/invariant remove <id>`                         | Удалить конкретное ограничение по ID          |
| `/invariant clear`                               | Удалить все ограничения                       |

Категории влияют только на отображение и группировку в `list` и в системном промпте.

## Зачем это нужно

### Проблема без инвариантов

Без механизма ограничений LLM-агент действует по намерению пользователя без проверки соответствия:

```
User: "Напиши на Python"     → LLM пишет на Python  (хотя проект на Kotlin)
User: "Используй Singleton"  → LLM использует Singleton  (хотя запрещён)
User: "Верни HTML"           → LLM возвращает HTML  (хотя нужен JSON)
```

При ротации участников команды или долгих сессиях системные ограничения легко забываются или не объясняются достаточно явно в промпте.

### Что даёт Invariant Management System

| Без инвариантов                     | С инвариантами                                    |
|-------------------------------------|---------------------------------------------------|
| LLM следует любому запросу          | LLM проверяет каждый ответ против правил          |
| Ограничения забываются при рестарте | Инварианты персистентны между сессиями            |
| Нет стандарта отказа                | Структурированный отказ с ID правила              |
| Пользователь не знает о запретах    | `/invariant list` показывает активные ограничения |
| Нет аудита                          | Каждое правило имеет ID для трассировки           |
| Ограничения в промпте теряются      | Блок инвариантов приоритетен над историей         |

### Примеры практического применения

**Технический стек:**
```
/invariant add --category stack Использовать только TypeScript, JavaScript запрещён
/invariant add --category stack Версия Node.js минимум 20 LTS
```

**Архитектурные решения:**
```
/invariant add --category architecture Запрещено использовать ORM — только raw SQL
/invariant add --category architecture Все сервисы строго stateless
```

**Бизнес-правила:**
```
/invariant add --category business-rule Цены отображать только в рублях
/invariant add --category business-rule Персональные данные не логировать
```

## Вывод

Invariant Management System превращает «мягкие рекомендации в промпте» в **жёсткие, проверяемые ограничения**. Ключевые характеристики:

1. **Персистентность**: инварианты переживают рестарты — хранятся в `~/.llmchat/invariants.json`
2. **Изоляция**: никогда не смешиваются с историей диалога, не вытесняются из контекста
3. **Приоритет**: инжектируются первыми в системный промпт — максимальный вес при генерации
4. **Аудируемость**: каждый отказ содержит ID правила — можно отследить, что именно заблокировало запрос
5. **Управляемость**: полный CRUD через CLI без перезапуска приложения

Ключевой инсайт: разница между `ctx: ~103` (нарушение с историей) и `ctx: ~0` (нарушение без истории) не меняет поведение — если инвариант есть в системном промпте, LLM откажет в любом случае. Это демонстрирует, что система **не зависит от контекстного окна**: инварианты не вытесняются скользящим окном истории.
