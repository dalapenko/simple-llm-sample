# Тематика
Планирование дня в ограниченных условиях

## Слабая модель (mistral-7b)
### Input:
У меня есть три задачи на сегодня:
- Встреча с клиентом (90 минут).
- Написание отчета (в 2 раза дольше, чем встреча).
- Обед (30 минут).
Условия:
- Мой рабочий день начинается в 10:00.
- Между задачами 1 и 2 должен быть перерыв 15 минут.
- Обед должен начаться строго после того, как будет готов отчет.
- Клиент может встретиться только в интервале с 11:00 до 14:00.
Задание: 
Рассчитай самое раннее время завершения всех задач. Ответ выдай строго в формате JSON с ключами: "schedule" (список задач с временем начала и конца), "total_duration_minutes" (общее время работы с учетом перерывов) и "finish_time" (время окончания последней задачи). Не пиши никакого вводного текста, только JSON.
### Output:
```jsong...
{
  "schedule": [
    {
      "task": "Встреча с клиентом",
      "start_time": "11:00",
      "end_time": "12:30"
    },
    {
      "task": "Перерыв",
      "start_time": "12:30",
      "end_time": "12:45"
    },
    {
      "task": "Написание отчета",
      "start_time": "12:45",
      "end_time": "16:45"
    },
    {
      "task": "Обед",
      "start_time": "16:45",
      "end_time": "17:15"
    }
  ],
  "total_duration_minutes": 495,
  "finish_time": "17:15"
}
```
### Statistics
- **Time:** 4.65с
- **Tokens:** 436
- **Price:** 0.000087$

## Вывод
### Анализ
#### Качество ответов (Logic & Reasoning)
Все три модели не справились с математической ловушкой идеально, но сделали это по-разному:

GPT-4o и Qwen 2.5 допустили идентичную ошибку: они выделили на отчет 150 минут (с 12:45 до 15:15), хотя по условию он должен длиться в 2 раза дольше встречи (90 мин × 2 = 180 мин). Однако они верно учли ограничение по времени начала встречи (11:00).

Mistral 7B полностью потеряла логику: она сделала отчет 4-часовым , а параметр total_duration_minutes (495) у нее вообще не совпадает с разницей между временем начала и конца. Это классический признак «галлюцинаций» слабых моделей в расчетах.

Qwen 2.5 проявила «самодеятельность», добавив задачу «Подготовка к встрече» с 10:00 до 11:00. В промте этого не было — модель пыталась заполнить логическую паузу в графике.

#### Скорость (Performance)
Здесь мы видим парадокс:

Самая сильная модель (GPT-4o) оказалась самой быстрой (2.77с). Это результат глубокой оптимизации инфраструктуры OpenAI и использования специфических архитектур.

Средняя модель (Qwen 72B) — самая медленная (7.10с). Это объясняется огромным количеством параметров (72 миллиарда), которые требуют колоссальных вычислительных мощностей для генерации каждого токена.

Mistral 7B показала средний результат (4.65с), что для модели такого размера на хорошем хостинге могло бы быть и быстрее.

####  Ресурсоёмкость (Cost-efficiency)
GPT-4o в 25 раз дороже, чем Mistral 7B. При этом для данной конкретной задачи (где она тоже ошиблась в расчетах) такая переплата может быть не оправдана.

Qwen 2.5 72B — «золотая середина». Она стоит почти как слабая модель, но по качеству логики стоит в одном ряду с GPT-4o. Это делает её лучшим выбором для сложных задач при ограниченном бюджете.

### Итог
Слабые модели (Mistral 7B) непригодны для задач, требующих многошаговой логики и арифметики. Они «плывут» в цифрах и не могут свести баланс времени.

Средние модели (Qwen 72B) сейчас максимально близки к лидерам по интеллекту, но могут проигрывать в скорости ответа.

Сильные модели (GPT-4o) обеспечивают лучший пользовательский опыт (скорость + формат), но их «интеллект» в математических задачах всё еще требует перепроверки (Chain-of-Thought).
