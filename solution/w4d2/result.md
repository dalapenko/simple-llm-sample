# Собственный MCP-сервер — JSONPlaceholder API

## Что сделано

Мы написали собственный MCP-сервер с нуля и подключили его к CLI. Сервер предоставляет агенту доступ к [JSONPlaceholder](https://jsonplaceholder.typicode.com/) — учебному REST API с постами и комментариями.

---

## Как это работает

```
CLI запускает MCP-сервер как дочерний процесс
          │
          ▼  stdin / stdout (JSON-RPC 2.0)
┌─────────────────────┐
│   mcp-server        │
│   JSONPlaceholder   │──── HTTP ────▶ jsonplaceholder.typicode.com
└─────────────────────┘
          │
     объявляет инструменты
          │
          ▼
    LLM видит инструменты и вызывает их в разговоре
          │
          ▼
    сервер делает HTTP-запрос, возвращает данные
          │
          ▼
    LLM формирует ответ пользователю
```

Транспорт — **stdio**: каждое сообщение это одна строка JSON, клиент и сервер общаются через стандартный поток ввода-вывода. Это самый простой способ запустить MCP-сервер — любая программа, умеющая читать stdin и писать в stdout, может стать сервером.

Все логи и отладочная информация идут в **stderr**, чтобы не засорять JSON-поток на stdout.

## Инструменты сервера

| Инструмент         | Параметры                 | Описание                                   |
|--------------------|---------------------------|--------------------------------------------|
| `get_posts`        | `userId` (необязательный) | Список постов, можно фильтровать по автору |
| `get_post_details` | `postId` (обязательный)   | Полное содержимое конкретного поста        |
| `get_comments`     | `postId` (обязательный)   | Все комментарии к посту                    |

Инструменты описаны по стандарту **JSON Schema** — LLM видит не только названия, но и типы параметров, какие из них обязательны, и текстовые описания. Это позволяет агенту самостоятельно решать, какой инструмент вызвать и с какими аргументами.

## Руководство по использованию

### 1. Сборка сервера

```bash
./gradlew :mcp-server-jsonplaceholder:shadowJar
```

Собирает `mcp-server-jsonplaceholder/build/libs/mcp-server-jsonplaceholder-1.0-SNAPSHOT-all.jar`.

### 2. Подключение из CLI

```
/mcp connect java -jar mcp-server-jsonplaceholder/build/libs/mcp-server-jsonplaceholder-1.0-SNAPSHOT-all.jar
```

CLI запускает JAR как дочерний процесс, выполняет MCP-хендшейк и передаёт список инструментов агенту.

### 3. Просмотр инструментов

```
/mcp tools
```

Покажет три инструмента с описаниями параметров.

### 4. Работа с агентом

После подключения можно общаться на естественном языке — агент сам решит, когда и какой инструмент вызвать:

```
Покажи посты пользователя с ID 1
Расскажи подробнее о посте с номером 3
Прочитай комментарии к посту 3
```

### 5. Отключение

```
/mcp disconnect
```

## Плюсы и минусы собственного MCP-сервера

### Плюсы

**Полный контроль.** Любой REST API, база данных или локальный инструмент превращается в MCP-сервер за несколько часов. Не нужно ждать, пока кто-то напишет готовый сервер.

**Никаких Node.js.** В отличие от большинства готовых серверов из экосистемы MCP, наш сервер — обычный JVM JAR. Нет зависимости от npm, нет скачивания при первом запуске.

**Понимание протокола.** Реализация JSON-RPC 2.0 вручную — хороший способ понять, как MCP работает изнутри. Это снимает «магию» с готовых решений.

**Обработка ошибок под контролем.** Сервер сам решает, что вернуть при 404, таймауте или некорректных параметрах. Можно форматировать ответы удобным для LLM образом.

### Минусы

**Больше кода.** Готовый сервер из npm-экосистемы подключается одной командой. Собственный требует написания и поддержки протокольного слоя.

**Нет авторизации.** Текущая реализация работает с публичными API без ключей. Добавление OAuth или API-ключей — дополнительная работа.

**Только локальный запуск.** Сервер запускается как дочерний процесс на той же машине. Для сетевого MCP (HTTP/SSE транспорт) нужна дополнительная реализация.

**LLM не всегда правильно сопоставляет запрос с инструментом.** Если описание инструмента неточное или запрос пользователя неоднозначный — агент может вызвать не тот инструмент или передать неверные аргументы. Качество описаний в JSON Schema напрямую влияет на качество работы агента.

## Архитектура модуля

```
mcp-server-jsonplaceholder/
├── Main.kt                   # точка входа: runBlocking { McpServer().run() }
├── McpServer.kt              # JSON-RPC 2.0 цикл, диспетчер методов, билдеры ответов
├── JsonPlaceholderClient.kt  # Ktor CIO HTTP-клиент, suspend-функции
└── Models.kt                 # Post, Comment — @Serializable data classes
```

Весь протокольный слой (~150 строк) написан на чистом `kotlinx.serialization.json` без сторонних MCP-библиотек.
