# Тематика

Демонстрация **Transition Guard System**: механизм принудительного соблюдения допустимых переходов между состояниями задачи в Task FSM. Новое состояние `PLAN_APPROVED` работает как обязательный контрольный пункт — агент не может перейти к реализации до явного подтверждения плана. Плагинный интерфейс `TransitionGuard` позволяет подключать произвольные бизнес-правила и инварианты к проверке переходов.

**Запись сессии**:
- [`demo.gif`](./demo.gif) — две последовательные сессии: попытка невалидного перехода → исправление маршрута → пауза → перезапуск → восстановление контекста

## Сценарий тестирования

**Идея**: показать полный жизненный цикл задачи с блокировкой недопустимого перехода `PLANNING → EXECUTION` (агент пытается перепрыгнуть через `PLAN_APPROVED`) и восстановлением состояния после перезапуска CLI.

**Модель**: GPT-4o Mini
**Стратегия**: Sliding Window (`--context-window 6`)
**Запуск**: `java -jar app.jar --strategy sliding-window --context-window 6`

### Задача демо

```
Разработать систему аутентификации пользователей
```

### Шаги сессии 1 — блокировка и исправление

```
 1. /task start Разработать систему аутентификации пользователей
 2. /task status                          ← начальное состояние: PLANNING
 3. [LLM] Как лучше хранить хэши паролей — bcrypt или argon2?
 4. /task advance execution               ← НЕВАЛИДНЫЙ переход PLANNING → EXECUTION
                                           → ERROR: Allowed from PLANNING: PLAN_APPROVED, ERROR
    ↺ REDO: "Невалидный переход заблокирован — исправляем маршрут"
 5. /task advance plan_approved           ← правильный переход PLANNING → PLAN_APPROVED
 6. /task status                          ← подтверждение: Stage: Plan Approved
 7. /task advance execution               ← теперь разрешён: PLAN_APPROVED → EXECUTION
 8. /task step Реализуем хэширование паролей через bcrypt
 9. /task pause                           ← сохранение в ~/.llmchat/task-state.json
    /exit
```

### Шаги сессии 2 — восстановление без переинтро

```
10. (CLI запускается) → баннер "Resuming Task" → ответ "y"
11. /task status                          ← Stage: Execution — состояние пережило перезапуск
```

## Механизм Guard — как это работает

### Граф переходов (новый)

```
PLANNING ──→ PLAN_APPROVED ──→ EXECUTION ──→ VALIDATION ──→ DONE
    │               │               │               │
    └──→ ERROR ←────┴───────────────┴───────────────┘
         │
         └──→ PLANNING
```

`PLAN_APPROVED` — обязательный контрольный пункт. Путь `PLANNING → EXECUTION` теперь **не существует** в графе. Это устраняет класс ошибок, при котором агент пропускал планирование и сразу приступал к реализации.

### Допустимые переходы

|       Из \ В        | PLANNING | PLAN_APPROVED | EXECUTION | VALIDATION | DONE | ERROR |
|:-------------------:|:--------:|:-------------:|:---------:|:----------:|:----:|:-----:|
|    **PLANNING**     |    —     |       ✅       |     —     |     —      |  —   |   ✅   |
|  **PLAN_APPROVED**  |    ✅     |       —       |     ✅     |     —      |  —   |   ✅   |
|    **EXECUTION**    |    ✅     |       —       |     —     |     ✅      |  —   |   ✅   |
|   **VALIDATION**    |    ✅     |       —       |     ✅     |     —      |  ✅   |   ✅   |
|      **DONE**       |    —     |       —       |     —     |     —      |  —   |   —   |
|      **ERROR**      |    ✅     |       —       |     —     |     —      |  —   |   —   |

`DONE` — терминальное состояние, переходов нет.

### Реализация

**`GuardResult`** — типобезопасный результат проверки:

```kotlin
sealed class GuardResult {
    data object Allow : GuardResult()
    data class Deny(val reason: String, val invariantId: String? = null) : GuardResult()
}
```

**`TransitionGuard`** — плагинный интерфейс (SAM, lambda-совместимый):

```kotlin
fun interface TransitionGuard {
    fun validate(from: TaskStage, to: TaskStage): GuardResult
}
```

**Порядок проверок в `TaskFSM.transition()`:**

```kotlin
fun transition(newStage: TaskStage, ...): Result<TaskState> {
    // 1. Структурная проверка графа
    val allowed = validTransitions[state.stage] ?: emptySet()
    if (newStage !in allowed) {
        return Result.failure(IllegalStateException("Invalid transition: ..."))
    }

    // 2. Бизнес-правила через guards
    for (guard in guards) {
        val result = guard.validate(state.stage, newStage)
        if (result is GuardResult.Deny) {
            return Result.failure(IllegalStateException(
                "Transition blocked: ${result.reason}" +
                result.invariantId?.let { " [Invariant: $it]" }.orEmpty()
            ))
        }
    }

    // 3. Мутация состояния и запись истории
    state = state.copy(stage = newStage, history = state.history + record)
    return Result.success(state)
}
```

Если переход заблокирован — состояние **не изменяется**, история **не обновляется**.

### Интеграция с инвариантами

Guards позволяют подключить `InvariantStorage` без прямой зависимости FSM от хранилища:

```kotlin
val guard = TransitionGuard { _, to ->
    if (to == TaskStage.PLAN_APPROVED && !planDocumentExists()) {
        GuardResult.Deny(
            reason = "Документ плана не создан",
            invariantId = "INV-A1B2C3"   // ссылка на конкретный инвариант
        )
    } else {
        GuardResult.Allow
    }
}
val fsm = TaskFSM.create("моя задача", guards = listOf(guard))
```

Сообщение об ошибке будет содержать как причину отказа, так и ID инварианта: `Transition blocked: Документ плана не создан [Invariant: INV-A1B2C3]`.

## Плюсы и минусы механизма Guard

### Плюсы

| Аспект                              | Описание                                                                                                       |
|-------------------------------------|----------------------------------------------------------------------------------------------------------------|
| **Compile-time безопасность**       | `sealed class GuardResult` исключает непроверенные исходы — компилятор требует обработки `Allow` и `Deny`      |
| **Декларативный граф**              | Переходы описаны в `Map<TaskStage, Set<TaskStage>>` — граф читается как таблица, а не как лабиринт `when`/`if` |
| **Расширяемость без изменений FSM** | Новые правила подключаются через `guards: List<TransitionGuard>` — FSM не требует модификации                  |
| **Ссылка на инвариант в ошибке**    | `Deny(reason, invariantId)` позволяет процитировать нарушенный инвариант прямо в сообщении об ошибке           |
| **Атомарность**                     | Заблокированный переход не мутирует состояние и не добавляет запись в историю                                  |
| **Персистентность**                 | История переходов сериализуется вместе с состоянием — маршрут задачи не теряется при перезапуске               |
| **Traceability**                    | Каждый переход фиксируется в `history[]` с меткой времени — можно восстановить весь путь задачи                |

### Минусы

| Аспект                                            | Описание                                                                                                                                                                                  |
|---------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| **Двойная проверка одного перехода**              | Структурный граф и Guards — два уровня валидации. Если Guard противоречит графу (разрешает структурно запрещённый переход), это невозможно — но сам факт двух уровней может запутать      |
| **Guards — stateless по умолчанию**               | `fun interface` принимает только `(from, to)` без доступа к текущему `TaskState` — для сложных правил (например, «минимум 3 перехода перед DONE») нужна closure или wrapper с состоянием  |
| **Нет приоритетов между Guards**                  | Первый `Deny` выигрывает — если два Guard конфликтуют, побеждает тот, что стоит раньше в списке. Неочевидный порядок может давать сюрпризы                                                |
| **PLAN_APPROVED нарушает обратную совместимость** | Существующие тесты, сценарии и сохранённые `task-state.json` с маршрутом `PLANNING → EXECUTION` стали невалидными — потребовалось обновление всех тестов                                  |
| **Дополнительный шаг пользователя**               | Каждый цикл работы над задачей теперь требует явного `/task advance plan_approved` — это осознанная нагрузка, но для автоматизированных сценариев без human-in-the-loop может быть лишним |

## Вывод

Transition Guard System превращает FSM из простой карты переходов в **управляемый рабочий процесс с enforcement-точками**. Ключевые характеристики:

1. **Обязательный `PLAN_APPROVED`** — структурно невозможно перепрыгнуть из планирования к реализации
2. **Расширяемые Guards** — любые бизнес-правила и инварианты подключаются без изменения FSM
3. **Атомарность** — заблокированный переход не оставляет следов в состоянии
4. **Персистентность маршрута** — история переходов выживает после перезапуска
5. **Traceability** — каждый переход (успешный или заблокированный) логируется

Ключевой инсайт из демо: `/task advance execution` в состоянии `PLANNING` возвращает явное сообщение `Allowed from PLANNING: PLAN_APPROVED, ERROR`. Агент получает не просто отказ, но и подсказку о корректном следующем шаге — это превращает ошибку FSM в обучающий сигнал.
